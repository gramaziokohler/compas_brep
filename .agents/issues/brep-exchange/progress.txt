Design point worth carrying into slice 02 (v6):
  The "plane" tag pins no parameterization — point + normal fix no x-axis
  and no domain — so a serialized pcurve on a planar face is meaningless to
  any reader. The builder needs a pcurve per trim, so planar faces re-derive
  theirs by projecting the edge curve onto the rebuilt plane (exact: the map
  is affine). OCC dodges this by ignoring pcurves for planes entirely and
  building from the 3D wire. Non-planar faces are fine — NURBS control points
  and the analytic types' COMPAS frames do pin the parameterization.
  ADR-0002 says v6 makes curve_2d non-nullable; for "plane" that is only
  meaningful if v6 also pins the plane's frame + uv_domain. Flagging it
  because ADR-0002 explicitly declined to adopt compas_rhino's frame/uv_domain.

THE TAG WAS THE EASY HALF. READ THIS BEFORE SLICE 05.
  A COMPAS CylindricalSurface is parameterized (angle, height), and OCC's
  pcurves already are. Rhino parameterizes a cylinder wall by ARC LENGTH: its
  native u is radius*angle, so r=0.5 runs u to pi for a full turn and r=0.3 runs
  it to 0.6pi. Emitting the cylinder tag with Rhino's pcurves untouched puts
  every trim at the wrong angle — a document worse than the nurbs tag it
  replaces, because it looks right and no face-count or tag test catches it.
  _cylinder_and_param_map recovers the map by probing, then VERIFIES it across
  the whole domain. Slice 05 must do the same for sphere/cone/torus and must not
  assume the map: measure it. Do not assume u is the angular direction either —
  on a fillet face u is axial and v is angular.

Measured, not assumed — a fillet face is a cylinder Rhino cannot tag:
  Rhino's 12 fillet faces on a filleted box ARE exactly cylinders to
  TryGetCylinder (at 1e-9), and OCC tags all 12 `cylinder`. They stay `nurbs`
  here: Rhino stores them as rational NURBS whose angle is NOT linear in either
  parameter (measured: second differences ~0.01 rad), so no affine map exists
  and the pcurves cannot be carried across exactly. `nurbs` reproduces them
  exactly — they are natively NURBS — so nothing is lost geometrically, only the
  analytic tag. A REAL remaining OCC/Rhino divergence. Closing it needs the
  parameterization question settled, which is the same question slice 02 raised
  for `plane` (frame + uv_domain in the document; ADR-0002 declined it). That
  decision is now overdue: it blocks fillets here, and slice 05 will hit it the
  moment a sphere/torus face is stored as a NURBS rather than a RevSurface.

TWO PRE-EXISTING CROSS-BACKEND DEFECTS, FOUND BY READING AN OCC DOCUMENT IN
RHINO — WHICH NOTHING HAD EVER DONE. Both are fixed here; both changed the
committed fixtures.
  1. curve_2d meant OPPOSITE things in the two writers. OCC writes a pcurve in
     its EDGE's direction and reads it that way; Rhino wrote it in the TRIM's.
     Measured on both: for a reversed trim, OCC's pcurve start is dist 0.0 from
     the edge start, Rhino's was dist 0.0 from the edge END. Rhino's own PLANAR
     rebuild already assumed edge-direction (it projects the edge curve, then
     reverses on is_reversed), so the non-planar writer was the lone dissenter.
     Edge-direction won.
  2. Rhino's pcurve DOMAIN was unrelated to its edge curve's. OCC requires the
     two to share a parameterization; Rhino does not (a trim's domain is its
     own). Rhino's circular seam pcurve ran (0, pi) while its edge curve ran
     (-pi, 0); OCC read that as an edge with no range and refused to sew the
     face at all — "Standard_Failure: Bnd_Box is void". _align_pcurve_to_edge
     remaps knots only (control points untouched), so it is exact.
  Together: NO RHINO-AUTHORED NON-PLANAR FACE HAS EVER BEEN READABLE BY OCC
  CORRECTLY. This is most of slice 03's "OCC flips the orientation of curved
  faces": the filleted box fixture goes 4.054412 -> 7.063681 on this change
  alone (true 7.563414). It is NOT all of it — filleted_box and sphere remain
  wrong and stay strict xfail, and slice 03 already proved an OCC-authored
  filleted box breaks the same way inside OCC. A separate OCC-side defect
  remains.

  0.0). It reads as something slice 04 broke. It is not. With the PRE-slice-04

2026-07-18 — Slice 05 (Rhino sphere/cone/torus surfaces): DONE
commit 31eb59d

Sphere, cone, torus now extract/write/read/rebuild as their COMPAS analytic
types through Rhino. All four analytic schema xfails gone on both backends
(RHINO_UNWRITABLE_SURFACE_TAGS is empty). Coverage now matches OCC.

THE PARAMETERIZATION IS STILL THE SLICE, AS SLICE 04 WARNED — but the shape
of the answer changed. Slice 04's _cylinder_and_param_map probed Rhino's
(u,v) and mapped to (angle,height) with a formula written in the Rhino
backend. That does not scale to four tags with four different v meanings.
Instead the document's parameter space is now stated ONCE, as an evaluator,
in exchange.py (analytic_surface_point / _params / _v_is_periodic): u is the
angle about the frame z for every tag; v is height (cylinder), latitude
(sphere), pipe-angle (torus), or slant distance from the base (cone). The
Rhino probe-then-verify map recovers native->document by calling that
evaluator, so it checks against the FORMAT, not against what this backend
believes the format is. _param_map_holds evaluates the same function rather
than a re-derived formula.

Measured, not assumed (the issue asked): Rhino parameterizes ALL of these by
arc length, confirmed at non-unit radii (r=1 is a trap — arc length == angle
there, so a sphere/torus looks affine and isn't distinguishable). The cone
convention genuinely differs: Rhino's plane origin is the APEX with z toward
the base; COMPAS is base-origin, apex at +height*z. _compas_cone_from_rhino
reverses it. OCC->Rhino->OCC keeps radius 0.5, height 1.0.

THE DOCUMENT'S PARAMETER SPACE IS OCC'S, AND THAT IS NOW PINNED ON CI.
test_exchange_parameterization.py evaluates every analytic tag against the
real OCP gp_ surface (Value(u,v)) at a tilted frame. If the evaluator ever
drifts from OCC, every pcurve silently means something else and nothing else
would catch it (writer and reader drift together). Needs OCC only.

REBUILD CANNOT DELEGATE TO RHINO'S CONSTRUCTORS. Measured: Sphere.ToRevSurface
happens to agree with the document, but Torus.ToRevSurface is arc-length in
BOTH directions and Cone.ToRevSurface measures v as height-from-apex, not
slant-from-base. Any of those lands every trim wrong. _analytic_surface_for_face
revolves the document's OWN generating curve (built from analytic_surface_point),
so it agrees by construction for all four. The cylinder's ToRevSurface path is
gone, replaced by this.

TWO PRE-EXISTING DEFECTS, BOTH FOUND BY READING AN OCC DOCUMENT IN RHINO —
which slice 04 did for the cylinder but NOT for pole/apex shapes. Both fixed.
  1. OCC and Rhino spell a collapsed boundary DIFFERENTLY. OCC writes a pole
     or apex as a zero-length ("degenerate") edge; Rhino writes a singular
     trim (edge -1, no edge curve) — the mechanism slice 01 built. Rhino
     cannot hold a degenerate edge (rejects the whole Brep: "Line points are
     coincident"). brep_to_rhino now detects a degenerate edge by LENGTH (not
     coincident endpoints — a closed circular seam has those too) and routes
     its trims to AddSingularTrim. Also had to apply is_reversed to singular
     trims: OCC reverses the pole trim, and a loop whose trims don't run head
     to tail in parameter space is not a loop ("do not match"). SPHERE AND
     TORUS ARE THE FIRST SHAPES OCC->RHINO HAS EVER PRODUCED AS VALID SOLIDS
     (slice 04 noted OCC->Rhino made a solid of nothing). Cone/cylinder come
     back valid-but-not-solid — the same planar-cap defect, not this.
  2. A planar cap's surface patch was sized from loop VERTICES. A circular cap
     loop has ONE vertex, on its rim. The patch was padded 1.0 around that one
     point, so a disc of radius > ~0.5 fell partly outside the patch and
     trimmed SMALL. The cylinder (r=0.5) squeaked under the pad; the cone
     (r=0.6) did not — its cap came back radius ~0.566, area 1.007 vs 1.131,
     volume 0.534 vs 0.565. Now sized from the boundary CURVES' control points
     (a rational circle's control polygon bounds the circle). Latent since
     slice 01; the cone is just the first face to cross the threshold.

A VOLUME/PARAM TRAP, measured not assumed: the sphere's "samples match at
matching parameters" check (slice 04's cylinder criterion) FAILED at first —
rebuilt point_at(0,0) was the opposite pole. Not a bug: a sphere has no
preferred pole, so Rhino's TryGetSphere picks an equatorial-plane orientation
arbitrarily and re-extraction can flip v. The two spheres are the SAME point
set (Hausdorff 4.9e-12 both ways), same radius/centre, round-trip valid +
solid + volume-exact. The surface-identity test is now a set comparison, not
pointwise (u,v); pcurve landing is verified by the valid+volume round-trip,
which is what a mislanded pcurve breaks. Cone and torus DO re-extract stably
(they have a real axis), but the test is uniform.

Fixtures: rhino_sphere flipped nurbs->sphere (its OCC rebuild volume now
