2026-07-16 — Slice 01 (Rhino rebuild via the low-level Brep builder): DONE
commit 2a9bf8c

Ported compas_rhino's _RhinoBrepBuilder to backend/rhino/builder.py and
rewrote brep_to_rhino to use it. Dropped _trim_nurbs_surface_from_2d, the
JoinBreps stitch, and the hand-tuned 1e-6 tolerance (builder runs at
TOL.absolute). Singular trims now survive: serialized as edge: -1 + vertex,
rebuilt via AddSingularTrim. BrepTrim gained .vertex / .is_singular.
Format stays v5.

Design point worth carrying into slice 02 (v6):
  The "plane" tag pins no parameterization — point + normal fix no x-axis
  and no domain — so a serialized pcurve on a planar face is meaningless to
  any reader. The builder needs a pcurve per trim, so planar faces re-derive
  theirs by projecting the edge curve onto the rebuilt plane (exact: the map
  is affine). OCC dodges this by ignoring pcurves for planes entirely and
  building from the 3D wire. Non-planar faces are fine — NURBS control points
  and the analytic types' COMPAS frames do pin the parameterization.
  ADR-0002 says v6 makes curve_2d non-nullable; for "plane" that is only
  meaningful if v6 also pins the plane's frame + uv_domain. Flagging it
  because ADR-0002 explicitly declined to adopt compas_rhino's frame/uv_domain.

Two of this slice's acceptance criteria did not discriminate — measured, not
assumed, by running both old and new code in live Rhino:
  - boolean-cut cylinder: the OLD path already passed (7/7 faces, exact
    volume). A through-cut cylinder wall IS a rectangle in parameter space,
    so the rectangular crop was adequate. The issue's claim that the old
    path failed here is wrong.
  - sphere IsValid: the OLD path also reported IsValid — ToBrep() regenerated
    an untrimmed sphere, so the dropped poles never surfaced. Validity does
    not pin this; the serialized singular-trim count (2) does.
  - filleted box: the ONLY case that caught the real defect. Old 26 -> 18
    faces, volume 7.563414 -> 7.063681. New: 26 -> 26, volume exact.
Future slices: prefer a filleted/non-rectangular face as the tracer geometry.

Fixed two pre-existing test faults the old code had been masking:
  - UNIT_BOX_OCC_DATA's bottom face wound against its own normal (exactly one
    flipped face -> volume 2/3). CreatePlanarBreps re-derived each face normal
    from its 3D boundary, so the bad winding never showed. Also note this
    fixture is hand-written and does NOT match real OCC output: OCC emits
    natural surface normals plus is_reversed: True, not outward-baked normals
    with all-False flags. Slice 03 should replace it with a real backend-
    authored fixture.
  - test_serialization_format_box_data_version asserted version == 4; the
    writer has emitted 5 for a release. Exactly the drift CONTEXT.md warns
    about, still sitting there because -m rhino never runs.

2026-07-16 — Slice 02 (v6: explicit loop roles, non-nullable pcurves): DONE
commit 8e6c33b

Loops are now tagged {"type": "outer"|"inner", "trims": [...]}; version is 6
from both writers (OCC tags via BRepTools.OuterWire_s, Rhino via BrepLoopType);
both readers pick the outer loop by tag and raise on zero or several. curve_2d
is non-nullable in v6 and the pcurve-presence branches are gone from both
rebuild paths (OCC's all_have_pcurves -> domain-bounded fallback, Rhino's two
None-guards).

New module src/compas_brep/exchange.py holds what the backends must agree on
(version, roles, loop decode, pcurve decode). Both legacy concessions are
confined there and nowhere else: the v4/v5 positional fallback and the null
pcurve those versions permitted.

Design point that cost the most thought — non-nullability is a property of a v6
DOCUMENT, not of the reader. The committed v4 fixture (test_rhino_serialization
UNIT_BOX_OCC_DATA) is planar-only with curve_2d: null on every trim, so a reader
that enforced non-null retroactively would fail the v4 read test this slice is
required to keep passing. trim_pcurve_from_data therefore raises only at v6.

Measured, not assumed:
  - The pcurve criterion cost nothing. Checked both writers first: OCC and Rhino
    each already emit a pcurve for every trim of a box / holed box / sphere.
    Zero nulls. The "writer that cannot produce a pcurve" is hypothetical on
    both backends today.
  - The inner-loop criterion caught a REAL pre-existing defect, and it was not
    in the format. Box with a through-hole never round-tripped on OCC: volume
    7.434513 -> 7.623009, restored shape invalid. Identical numbers before the
    slice, so v6 didn't cause it. Cause: a planar face rebuilds from its 3D
    wires; a 3D wire's winding says nothing about hole-ness, OCC reads that from
    wire orientation, and MakeFace(face, wire) does not reorient. One of the two
    holed faces got an inner wire winding with its outer, so OCC ADDED the hole
    area (4.283 = 4 + pi*0.3^2) instead of subtracting it (3.717). Only one of
    the two broke — top and bottom have opposite orientations — which is how a
    symmetric shape hid it. ShapeFix_Face.FixOrientation fixes it; delta now
    4.5e-08 (so the test asserts TOL.is_close, not TOL.absolute — 1e-9 is below
    the rebuild's own sewing tolerance).

Slice 01's flagged design point, now resolved as far as this slice can:
  Planar pcurves stay WRITE-ONLY. Routing planar faces through the pcurve path
  (which would delete the 3D-wire path entirely) does not work, because
  Plane.__data__ is point + normal and pins no x-axis — a reader re-deriving the
  plane picks an arbitrary UV frame and the pcurve lands elsewhere. OCC ignores
  planar pcurves; Rhino re-derives its own by projection. v6 makes them
  non-nullable but not meaningful. Making them meaningful needs the plane's
  frame + uv_domain pinned in the document — which ADR-0002 explicitly declined.
  Worth a decision (and an ADR amendment) before anyone relies on them.

Carried forward — KNOWN DEFECT, not in any slice-02 criterion:
  A cylinder loses its SEAM EDGE on rebuild and the result reports invalid.
  Plain cylinder: 3 edges -> 2, BRepCheck_Analyzer invalid. Holed box: 15 -> 14.
  Pre-existing (identical before this slice). It survives because face count and
  volume — what the criteria check — are both unaffected by it. Slice 06
  (analytic edge tags: circle/arc/ellipse) touches seam curves and is the
  natural home for it.

CONTEXT.md drift, left alone deliberately: its v6 section already claims edge
curves carry analytic tags (line | circle | arc | ellipse | nurbs) and that
"both backends read and write every tag". That is slice 06, not yet true.
